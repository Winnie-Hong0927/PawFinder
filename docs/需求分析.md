# PawFinder 宠物领养系统需求分析

## 1. 项目概述

### 1.1 项目背景
PawFinder 是一个面向宠物领养场景的综合平台，连接宠物救助机构与有领养意向的用户，通过严格的审核流程确保领养过程的透明和安全。

### 1.2 项目目标
- 为流浪宠物提供展示平台，帮助它们找到合适的领养家庭
- 为宠物救助机构提供便捷的宠物管理和领养审核工具
- 为普通用户提供便捷的宠物浏览和领养申请渠道

### 1.3 目标用户
| 用户类型 | 描述 |
|---------|------|
| 普通用户 | 有领养意向的个人，可在平台浏览宠物、提交领养申请 |
| 机构管理员 | 各宠物救助机构的管理人员，负责本机构宠物的管理和领养审核 |
| 系统管理员 | 平台运营方，负责机构接入审核和机构管理员注册审核 |

---

## 2. 用户角色与权限

### 2.1 角色说明

| 角色 | 权限范围 |
|------|---------|
| **普通用户 (user)** | 浏览宠物、收藏宠物、提交领养申请、管理个人信息 |
| **机构管理员 (institution_admin)** | 管理本机构宠物、审核本机构领养申请、查看申请人信息 |
| **系统管理员 (sysadmin)** | 审核机构接入申请、审核机构管理员注册、管理所有宠物和申请 |

### 2.2 权限矩阵

| 功能 | 普通用户 | 机构管理员 | 系统管理员 |
|------|:--------:|:----------:|:----------:|
| 浏览宠物列表 | ✓ | ✓ | ✓ |
| 查看宠物详情 | ✓ | ✓ | ✓ |
| 收藏宠物 | ✓ | - | - |
| 申请领养 | ✓ | - | - |
| 添加/编辑本机构宠物 | - | ✓ | ✓ |
| 删除本机构宠物 | - | ✓ | ✓ |
| 审核领养申请 | - | ✓(本机构) | ✓(全部) |

| 审核机构管理员注册 | - | - | ✓ |

## 3. 功能需求

### 3.1 用户模块

#### 3.1.1 用户注册与登录
- **登录方式**：手机号 + 短信验证码
- **自动注册**：首次登录时自动创建用户账号
- **角色分配**：登录后根据用户角色显示不同功能入口

#### 3.1.2 个人信息管理
- 修改昵称、头像
- 修改联系方式
- 查看申请记录

### 3.2 宠物模块

#### 3.2.1 宠物浏览（用户端）
- 宠物列表展示（支持筛选：种类、性别、体型）
- 宠物详情页（多图展示、基本信息、健康状况、性格特点）
- 收藏/取消收藏宠物

#### 3.2.2 宠物管理（机构管理员/系统管理员）
- 添加新宠物（上传多张照片、基本信息、健康状态）
- 编辑宠物信息
- 上架/下架宠物
- 删除宠物

### 3.3 领养申请模块

#### 3.3.1 申请提交（用户端）
- 填写领养申请表单
  - 领养理由
  - 居住环境
  - 养宠经验
  - 身份证号（实名认证）
- 查看申请状态

#### 3.3.2 申请审核（机构管理员）
- 查看申请人信息
- 查看申请人提交的领养资料
- 通过/拒绝申请
- 填写审核意见

### 3.4 机构管理模块（系统管理员）

#### 3.4.1 机构接入审核
- 查看待审核机构列表
- 查看机构详细信息（名称、描述、联系方式、营业执照）
- 通过/拒绝机构接入申请

#### 3.4.2 机构管理员注册审核
- 查看待审核管理员申请列表
- 查看申请人信息（姓名、电话、身份证）
- 通过/拒绝管理员注册申请
- 通过后自动创建机构管理员账号

## 4. 业务流程

### 4.1 用户领养流程

```
用户浏览宠物 → 点击申请领养 → 填写申请表单 → 提交申请
                                            ↓
机构管理员审核 → 通过/拒绝 ← 机构管理员查看申请信息
      ↓
审核通过 → 用户可上传领养后视频 → 完成领养
```

### 4.2 机构入驻流程

```
机构提交入驻申请 → 系统管理员审核 → 通过/拒绝
                                        ↓
通过 → 机构管理员注册申请 → 系统管理员审核 → 创建机构管理员账号
                                                      ↓
                                              机构管理员上传宠物信息
```

## 5. 数据模型

### 5.1 核心实体

| 实体 | 说明 |
|------|------|
| User | 用户信息（普通用户、管理员） |
| Institution | 宠物救助机构 |
| InstitutionAdminRequest | 机构管理员注册申请 |
| Pet | 宠物信息 |
| AdoptionApplication | 领养申请 |
| Video | 领养后回访视频 |

### 5.2 实体关系

```
User 1 ──N Institution (一个机构有多个管理员)
Institution 1 ──N Pet (一个机构发布多个宠物)
Institution 1 ──N AdoptionApplication (一个机构接收多个申请)
User 1 ──N AdoptionApplication (一个用户提交多个申请)
Pet 1 ──N AdoptionApplication (一个宠物有多个申请)
AdoptionApplication 1 ──N Video (一个申请有多个回访视频)
```

## 6. 非功能需求

### 6.1 性能需求
- 页面加载时间 < 3秒
- API 响应时间 < 1秒
- 支持并发用户数 > 100

### 6.2 安全需求
- 用户敏感信息加密存储
- 短信验证码时效性（5分钟）
- 管理员操作日志记录
- 机构认证材料验证

### 6.3 可用性需求
- 支持主流浏览器（Chrome、Safari、Firefox、Edge）
- 移动端适配（响应式设计）
- 图片懒加载优化

## 8. 图片需求说明

### 8.1 是否需要图片？

**技术文档中的图片（UML图等）不是必需的**，但对于复杂系统的需求分析，以下图片可以增强文档的可读性和沟通效率：

| 图片类型 | 是否建议 | 原因 |
|---------|:--------:|------|
| **用例图** | ✓ 建议 | 清晰展示系统与用户的交互边界 |
| **时序图** | ✓ 建议 | 展示核心业务流程的时序关系 |
| **ER图/数据模型图** | ✓ 建议 | 展示数据库表关系 |
| **页面原型图** | ✓ 建议 | 提前确认UI设计方向 |
| **活动图/流程图** | ✓ 建议 | 展示业务流程分支 |
| **类图** | ✗ 非必需 | 仅在需要详细设计时使用 |

### 8.2 图片制作优先级

**如果时间有限，建议优先制作：**

1. **用例图** - 用最少的篇幅说明系统边界
2. **页面原型图** - 最重要，直接影响开发方向
3. **业务流程图** - 帮助开发团队理解业务逻辑
